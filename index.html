Job Description

As a participative, hands-on member of the Software Quality Assurance team, I was responsible for executing and maintaining manual test scripts for different applications, performing manual regression testing, as well as preparing the required documentation.

Fitting Software

Hearing aids are amazing, complex devices capable of a plethora of different features. In order to adjust these features to the patient's needs, fitting software is used. For example, there is a feature called tinnitus masker which generates and emits broad-band or narrow-band noise at low levels, designed to mask the presence of tinnitus in a patient's ears. The fitting software would be used to adjust how low that level should be and at what frequency range. Another more simple example is toggling the startup sound delay of the hearing instrument. My job was to make sure that all of these functionalities within the fitting software works properly as intended. If I tested to make sure the tinnitus masker was actually working, I would take a sample hearing instrument, run the software for it, and check to ensure the tinnitus masker feature works as intended when I adjust its input values.

Testing Methodologies

An important concept that I learned in university is testing code and software. It is a crucial part of the Software Development Lifecycle, but I didn't recognize its importance until I did this work term. There are various documents I had to read called test scenarios and test cases. Test cases are procedures of individual steps that assess each feature of an application. A test scenario is a collective set of test cases which makes up a specific functionality of the software. For example, if I were doing a test case about the GUI of the software, I would verify that the appropriate icons, text and buttons appear as specified in the case, and then I would make sure these buttons and icons work properly when I click adjust or otherwise change them.

As for how these cases are actually conducted, there are several methods to test a piece of software. Generally speaking, the mindset of the software tester is "How can I break this piece of software?" Good to know because I'm much better at breaking things than building them. Simply testing that a software gives the expected output is not nearly enough to rigorously verify that the software is functioning properly. Just because it works doesn't mean there's potential bugs in it. Software testing can generally be divided into two types: functional and non-functional testing.Functional testing ensures that the functions and features of the application work properly. Non-functional testing examines other aspects of how well the application works. Functional testing tests the functionality of an app. Non-functional testing tests the performance of these functions. Maintenance testing is usually re-running functional and non-functional tests to ensure that previously developed and tested software still performs as expected after a change. Most of the testing I conducted fell into this category, specifically regression testing. I tested older versions of the fitting software to make sure they still ran smoothly and if there were any bugs, I would report them. A bug is when a feature or piece of code does not work as intended, and it must be reported in order to have it fixed by a developer.

Once a test is conducted, a document called a test report is filled up to specify how the process went. If the test failed, usually it is due to the presence of a bug and that bug also gets documented in a bug report. Whether it passes or fails, the test will be viewable on whatever website or tool the company uses to record documents. In Sonova's case, I worked with an Application Lifecycle Management website called Polarion.
